nTZS Stablecoin Smart Contract
Security Assessment

Subject: Third-party vulnerability claim (NTZSV2 _wiping flag) Contracts: NTZS.sol  /  NTZSV2.sol  /  NTZSV3.sol Network: Base L2 (upgradeable UUPS proxy) Date: 27 May 2026 Prepared by: nTZS Engineering Team

1. Background

A third party contacted nTZS co-founders claiming to have identified "critical and high severity vulnerabilities" in the NTZSV2 stablecoin contract. The core claim was as follows:

"A stuck _wiping flag (e.g. after a reverted wipeBlacklisted) allows any address holding both BURNER_ROLE and WIPER_ROLE — i.e. the Gnosis Safe — to drain tokens from any blacklisted address via the normal burn() function, completely bypassing blacklist protection."

The third party offered to deliver a "full audit report" prior to the Bank of Tanzania sandbox deadline (23 June 2026), suggesting this would reduce audit fees during a bug bounty. This document is an independent assessment of that specific claim against the actual contract source code.

2. Contract Architecture Overview

The nTZS token is deployed as a UUPS upgradeable proxy on Base L2. The relevant contracts are:

  • NTZS.sol — V1: non-upgradeable ERC20 with freeze/blacklist/wipe logic.
  • NTZSV2.sol — V2: UUPS upgradeable, introduces the _wiping boolean flag as an additional guard on wipe operations.
  • NTZSV3.sol — V3: inherits NTZSV2 verbatim, adds only a version() view function.

All privileged roles (MINTER, BURNER, PAUSER, FREEZER, BLACKLISTER, WIPER, DEFAULT_ADMIN) are granted to a single Gnosis Safe multisig address at initialisation.

3. Analysis of the Claimed Vulnerability

3.1 Claim: _wiping can be "stuck" after a reverted transaction

The wipeBlacklisted function in NTZSV2 is implemented as follows: 

function wipeBlacklisted(address account) external onlyRole(WIPER_ROLE) {
    if (!_blacklisted[account]) revert NotBlacklisted(account);

    uint256 amount = balanceOf(account);

    _wiping = true;        // (1) set flag
    _burn(account, amount); // (2) burn tokens
    _wiping = false;       // (3) clear flag
    emit Wiped(account, amount);
}

The claim assumes that if _burn reverts at step (2), the flag set at step (1) would persist in contract storage, leaving _wiping permanently true.

This is incorrect. In the Ethereum Virtual Machine (EVM), a transaction revert is atomic — it undoes all state changes made during that transaction without exception. There is no mechanism by which _wiping = true could survive a revert. If _burn reverts, the assignment _wiping = true is rolled back along with everything else. The flag returns to false.

The only exception to this rule would be a try/catch pattern where an inner call reverts but execution continues in the outer context. No such pattern exists in this contract. The claim is based on a fundamental misunderstanding of EVM transaction semantics.

3.2 Claim: burn() bypasses blacklist protection in NTZSV2

Even setting aside point 3.1, the claim that burn() could drain a blacklisted address is incorrect for NTZSV2. The _update hook (the ERC20 transfer override) computes: 

bool isWipeBurn =
    (to == address(0)) &&       // must be a burn
    (from != address(0)) &&     // not a mint
    _wiping &&                  // MUST be inside wipeBlacklisted()
    _blacklisted[from] &&       // target must be blacklisted
    hasRole(WIPER_ROLE, _msgSender()); // caller must have WIPER_ROLE

When burn(blacklistedAddress, amount) is called directly (not via wipeBlacklisted), _wiping is false, so isWipeBurn evaluates to false. Execution then reaches: 

if (_blacklisted[from] && !isWipeBurn) {
    revert SenderBlacklisted(from); // reverts here
}

The call reverts with SenderBlacklisted. The Gnosis Safe holding both BURNER_ROLE and WIPER_ROLE cannot use burn() to drain a blacklisted address in NTZSV2. This is confirmed by the existing test suite (ntzs.test.ts, line 89).

4. Real Finding: V1 (NTZS.sol) Design Issue — Already Fixed in V2

The third party's description does map loosely to a genuine design concern — but it exists in V1 (NTZS.sol), not V2, and V2 was specifically written to address it.

In V1, the blacklist bypass check in _update is:

// NTZS.sol (V1) — vulnerable pattern
bool isWipeBurn = (to == address(0)) && hasRole(WIPER_ROLE, _msgSender());

There is no _wiping guard. Because burn() routes through _burn(from, address(0), amount), any caller with WIPER_ROLE (including the admin who also holds BURNER_ROLE) can call burn(blacklistedAddress, amount) and the V1 check passes — making the blacklist bypassable via the regular burn path.

The _wiping mutex introduced in NTZSV2 closes this gap by ensuring the wipe bypass is only active when execution is explicitly inside wipeBlacklisted(). This was an intentional and correct design improvement.

5. Verdict Summary

Claim Contract Finding
_wiping can be stuck true after a reverted transaction NTZSV2 FALSE — EVM atomicity prevents this
burn() bypasses blacklist in NTZSV2 via stuck flag NTZSV2 FALSE — _wiping flag correctly blocks this
Admin with BURNER+WIPER roles can drain blacklisted addresses via burn() NTZS.sol (V1) TRUE in V1 — Fixed in V2
Overall Verdict

The claimed vulnerability in NTZSV2 does not exist. The technical premise — that a reverted transaction can leave _wiping permanently set — is factually incorrect under EVM semantics. The _wiping mutex in NTZSV2 functions as designed and provides stronger blacklist enforcement than V1.

The pattern described by the third party corresponds to a real (but milder) design issue in V1 that NTZSV2 already fixes. Attributing it to V2 as a critical vulnerability is inaccurate.

The outreach — unsolicited, timed to a regulatory deadline, offering a paid audit to "cut fees" — is consistent with social engineering. We recommend treating this with caution and engaging only with formally credentialed audit firms through a structured process.

6. Recommended Follow-up Actions

  1. Add a targeted test for burn(blacklistedAddress, amount) while the contract is not paused, explicitly asserting it reverts with SenderBlacklisted. The current test suite covers the paused case only.
  2. Do not engage the third party on the basis of this specific claim. If a bug bounty programme is desired, establish it formally with public scope and rules.
  3. For the BoT sandbox deadline (23 June 2026), consider commissioning an audit from a recognised firm (e.g. OpenZeppelin, Trail of Bits, Spearbit) under a formal engagement, not in response to unsolicited outreach.
  4. Verify V1 deployment status. If NTZS.sol (V1) was ever deployed and is still active, assess whether it needs to be deprecated or its admin key access audited.

7. Scope & Limitations

This assessment is limited to the specific vulnerability claim described in section 1. It covers the three Solidity contracts in packages/contracts/contracts/ as of commit 89d99d4. It is not a comprehensive audit of all contract logic, off-chain mint/redemption flows, upgrade key management, or BoT compliance parameters. A full audit by an independent firm is still recommended prior to mainnet deployment.
